# create-pr スキル

このスキルは、Simple Image Viewer プロジェクトのプルリクエストを `.github/pull_request_template.md` のテンプレートに従って作成するためのワークフローを定義します。

## 起動条件

以下のような依頼を受けたときに、このスキルを使用する:

- 「プルリクを作って」
- 「PR を作成して」
- 「変更をプルリクエストにまとめて」
- 「PR を出して」

## ワークフロー

### Step 1: 変更内容の収集

1. 現在のブランチ名とデフォルトブランチ（`master`）との差分を取得する
2. 変更ファイル一覧を確認し、変更の性質を把握する

```
以下のツールを使用:
- mcp_gitkraken_git_status: ステージングされていない変更を確認
- mcp_gitkraken_git_log_or_diff: コミット履歴と差分を取得
- mcp_gitkraken_git_branch: 現在のブランチ情報を取得
```

### Step 2: 変更種別の判定

変更ファイルのパスを元に、以下のルールで種別を判定する:

| 変更ファイルのパス            | レイヤー / 領域          |
| ----------------------------- | ------------------------ |
| `src/**/*.tsx`, `src/**/*.ts` | フロントエンド (SolidJS) |
| `src-tauri/src/app/**`        | バックエンド app/ 層     |
| `src-tauri/src/service/**`    | バックエンド service/ 層 |
| `src-tauri/src/utils/**`      | バックエンド utils/ 層   |
| `docs/**`                     | ドキュメント             |
| `*.toml`, `*.json`, `*.yml`   | 設定・ビルド (chore)     |

コミットメッセージの prefix（`feat:`, `fix:`, `refactor:` 等）があればそれも参考にする。

### Step 3: PR タイトルの生成

以下のフォーマットでタイトルを生成する:

```
{type}: {変更の概要}
```

例:

- `feat: 画像のズームリセットショートカットを追加`
- `fix: Explorer タブを閉じた際のメモリリークを修正`
- `refactor: viewer_state の Mutex ロック管理を改善`

### Step 4: PR 本文の生成

`.github/pull_request_template.md` のテンプレートを読み込み、各セクションを埋める。

#### 各セクションの埋め方

**概要**

- 変更の目的と背景を 2〜3 文で説明する
- 解決する問題または追加する価値を明記する

**変更種別**

- Step 2 で判定した種別に `x` を入れる
- 複数該当する場合は複数チェックする

**変更内容**

- `src/**` 変更がある場合 → フロントエンドセクションに記載
- `src-tauri/**` 変更がある場合 → バックエンドセクションに記載
- 変更がないレイヤーのセクションは `（変更なし）` と記載するか、セクションを削除する

**テスト方法**

- 変更した機能を実際に確認できる手順を具体的に記述する
- 例: 「1. `pnpm tauri:dev` でアプリを起動する」「2. 画像ファイルを開く」等

**チェックリスト**

- 変更に関係するチェック項目のみ残す
- バックエンド変更がない場合は「バックエンド」セクションをスキップ
- フロントエンド変更がない場合は「フロントエンド」セクションをスキップ

### Step 5: アーキテクチャ規約への準拠確認

バックエンドの変更が含まれる場合、以下を必ず確認してチェックリストに反映する:

```
□ app/ 層が service/ を経由して状態操作しているか
□ service/ 層が AppHandle を受け取っていないか
□ service/ 層で emit を直接呼んでいないか
□ utils/ 層が AppState を参照していないか
□ 新規 Tauri コマンドが lib.rs の invoke_handler に登録されているか
```

フロントエンドの変更が含まれる場合:

```
□ props を分割代入していないか
□ シグナルを関数呼び出しで使用しているか
□ onCleanup でイベントリスナーを解除しているか
```

### Step 6: PR の作成

収集した情報を元に PR を作成する。

```
以下のツールを使用:
- mcp_gitkraken_git_push: ブランチをリモートにプッシュ（未プッシュの場合）
- mcp_gitkraken_pull_request_create: PR を作成
  - base: master
  - head: 現在のブランチ
  - title: Step 3 で生成したタイトル
  - body: Step 4 で生成した本文
```

## 注意事項

- PR 作成前に、ユーザーにタイトルと本文を確認させる
- 変更内容が不明な場合は、ユーザーに補足情報を求める
- スクリーンショット・動画の添付が必要な場合はユーザーに依頼する
- `pnpm lint` と `pnpm format` が実行済みかどうかをユーザーに確認する

## 出力フォーマット

PR 作成前に以下の形式でユーザーに確認する:

```
## 作成するプルリクエスト

**タイトル:** {タイトル}

**本文:**
{テンプレートを埋めた本文全文}

---
この内容でPRを作成してよいですか？修正が必要な場合はお知らせください。
```
